COTAMS Frontend Specification (UI/UX + Pages + Role Flows)
1) Goal of the Frontend

The frontend is a role-based web application for a college timetable system. It must:

Work well on mobile, tablet, laptop, and desktop

Provide a clean, professional UI

Support light/dark theme

Use REST APIs for data

Use WebSockets to receive real-time “timetable updated” events and show in-app notifications

Enforce role-based navigation: users see only what their role allows

2) Tech Stack (Frontend)

React + Vite + TypeScript

Tailwind CSS

shadcn/ui components

Framer Motion for subtle transitions

Axios (with JWT interceptors)

React Router for routing (recommended)

Zustand (optional) for auth/token and UI state (recommended if you already planned it)

3) Design System (Visual Requirements)

Brand / colors

Primary: #0E21A0 (main buttons, links, active states)

Light theme base: white + zinc surfaces (soft, not too strong)

Card background (light): zinc-toned surface

Dark theme background: #30364F with zinc/dark surfaces

Typography: clear hierarchy, good spacing, no clutter

Responsiveness

Mobile-first layout

Sidebar on desktop, drawer navigation on mobile

Buttons and form inputs scale properly (full width on mobile)

Timetable grid switches to “day view” on small screens

4) Global App Layout (Authenticated Area)

After login, the app uses a shared layout shell:

Desktop layout

Left sidebar (fixed): navigation menu with icons

Top bar: page title/breadcrumb + theme toggle + notifications bell + user menu

Main content: responsive container with cards and grids

Mobile layout

Sidebar becomes a hamburger menu drawer

Top bar stays visible

Content becomes single-column stacked layout

Global components

AppShell

Sidebar + MobileDrawer

Topbar

ThemeToggle

NotificationsBell (shows unread count)

UserMenu (profile, logout)

7) Timetable Page (Main UI Feature)

The Timetable page is different by screen size and role.

Desktop Timetable View (weekly grid)

Layout:

Main area: Weekly grid (Mon–Fri or Mon–Sun)

Side area: Filters + summary

Grid design:

Columns = days

Rows = time range (07:00 to 21:00)

Each session is displayed as a card block spanning its time

Session card shows:

module code (bold)

room code

time range

optional lecturer name

Clicking a session opens:

a modal or side panel with full details

Filters (right side)

Academic year + semester

Stream (admin can switch)

Lecturer (admin filter)

Room (admin filter)

Mobile Timetable View (day-based)

On small screens, the weekly grid is hard to read, so show:

A day selector (tabs or dropdown)

A list of sessions for that day as cards

Each card shows same details, click to open modal

8) Session Management (Admin Only)
Admin “Create/Edit Session” page

This page is where the “real-time clash detection” is visible.

Form fields

Stream

Module

Lecturer

Room

Day of week

Start time

End time

Academic year + semester

UX behavior

On “Save session”:

call REST create endpoint

if clash: show detailed clash panel (do not save)

if success: show toast “Session saved” and return to timetable

Clash display

Show an error card explaining:

conflict type (room / lecturer / stream)

conflicting session details (time + module + room)

Optional: highlight the conflicting slot on timetable preview

10) Notifications UI (All Roles)

Two parts:

A) Real-time toast alerts

When a WebSocket event is received:

Show toast: “Timetable updated”

Update unread notification count

If user is currently on timetable page:

auto-refresh via REST or show “Refresh” button

B) Notifications Inbox Page

List notifications from the Notification table (REST)

Mark read / mark all read

Each notification shows:

title/message

timestamp

optional “View timetable” action

Important rule
WebSocket only delivers the event. The frontend should fetch the latest timetable using REST to stay consistent.

11) Audit Logs UI (Admin Only)

Audit logs must be visible to admin:

Table list:

time

actor user

action (create/update/delete)

entity (session id)

Details view shows:

before vs after changes (readable JSON)
This is for accountability (“who changed the timetable?”).

12) Data Fetching Rules (Frontend)

Use Axios instance with base URL from .env (VITE_API_BASE_URL)

Attach JWT access token to requests

Use /me endpoint on startup to load:

user identity

roles

permission codes (optional)

Use role-based rendering:

hide admin-only pages if not allowed

On WebSocket event:

show toast

refresh relevant data via REST
